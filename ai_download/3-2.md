## 0. 结论先行

### 0.1 三大审查结论

本次评审团不再沿用“从旧 PRD 理想态倒推”的方式，而是直接从当前仓库的真实状态出发，得出以下三条结论：

1. **当前 demo 的产品方向是对的，但它仍是“可演示原型”，不是“可上线 MVP”。**
   - 方向正确：绘本感、地图主舞台、时间轴驱动、Intro、局部聚焦、故事化面板已经存在。
   - 不能直接上线：核心媒体资产缺失、移动端面板未闭环、数据可信度说明不足、版权与引用链未闭环。

2. **PRD v3.1 的最大问题不是细节 bug，而是范围混叠。**
   - 它把“当前 demo 打磨”“在线数据平台”“运行时 AI 内容”“扩展物种系统”“环境音系统”混成一份执行稿。
   - 这会让 agent 同时面对 UI 打磨、数据工程、后端安全、内容合规、部署策略五条战线，MVP 边界被冲散。

3. **expert review 很有价值，但不能原封不动照搬。**
   - 它准确拦截了很多 v3.1 内部矛盾、伪代码错误、合规风险和部署漏洞。
   - 但它也有明显盲点：对当前代码库已实现能力利用不够，部分批评针对的是 v3.1 里“尚未真的进入仓库”的设计。

### 0.2 本版 PRD 的核心策略

`PRD v3.2` 采取以下策略：

- 以当前 demo 为唯一现实基线，优先把现有体验打磨成可交付的科普互动网页。
- 把核心 18 种物种的**本地化、可审校、可归因、可展示**体验定义为当前 MVP。
- 把“扩展物种 + ALA 在线查询 + 运行时 LLM + 地貌环境音自动切换”整体降级为**增强轨**，不得阻塞主版本。
- 不再用大段脆弱伪代码驱动 agent；改用更稳固的“文件职责 + 数据契约 + 阶段目标 + 验收标准”驱动开发。

### 0.3 审慎评分

| 审查对象 | 体验质量 | Agent 可执行性 | 上线可控性 | 结论 |
|---|---:|---:|---:|---|
| 当前 demo 代码 | 7/10 | 6/10 | 4/10 | 方向对，但媒体与发布闭环未完成 |
| PRD v3.1 | 5/10 | 7/10 | 3/10 | 过度雄心，细节多但过脆、范围过大 |
| expert review | 6/10 | 8/10 | 6/10 | 有效拦截风险，但对仓库现实贴合度不足 |

## 1. 竞争性审慎评审团

本评审团在原 expert review 的角色基础上重组，并补入更贴近当前项目的职责划分。所有结论以“当前 demo 是否能被安全、优雅、高效地演进”为准。

### A — 首席代码架构师
- 结论：当前 demo 的架构方向可保留，不应推倒重来。
- 关键判断：`AtlasContext + page-level composition + local JSON + MapLibre` 已形成基本骨架，`v3.1` 最大问题是强行并入多条异步在线数据流。
- PRD3.2 指令：保留现有前端骨架；先做“本地可靠 MVP”，再开“在线增强轨”。

### B — 前端代码工程师
- 结论：当前代码质量高于 v3.1 文档所假定的起点。
- 关键判断：仓库已有 `IntroSequence`、搜索下拉、地图 loading 态、selected marker portal、键盘可操作时间轴、reduced motion 接入，`build` 与 `lint` 当前均通过。
- PRD3.2 指令：前端优先修真实缺口，不先追逐 v3.1 那套尚未入库的复杂云层和扩展物种架构。

### C — 前端交互工程师
- 结论：当前 demo 的交互主线成立，但“层级过渡”与“移动端闭环”不足。
- 关键判断：全国态、区域态、物种态切换已存在，但面板在手机上没有收敛为底部抽屉；时间轴、图例、顶部栏在窄屏协同时仍有风险。
- PRD3.2 指令：先完成交互闭环和跨端体验，再增加新能力。

### D — UI/UX 设计师
- 结论：当前 demo 已有明确风格，但“规格化不足”仍成立。
- 关键判断：色彩、字重、纸张质感、panel 语言已具备，但视觉规则没有沉淀成 agent 可执行规范，导致后续新增 UI 极易跑偏。
- PRD3.2 指令：为现有组件补齐布局、间距、尺寸、状态、空态和过渡规范。

### E — 交互设计师
- 结论：当前站点已有故事节奏，但“引导层”还不够强。
- 关键判断：Intro 存在，这是 expert review 中需要修正的一点；但 Intro 与地图加载、搜索、面板切换之间缺乏统一的状态设计。
- PRD3.2 指令：保留 Intro，并把它和地图首屏、搜索、Panel 切换纳入同一交互剧本。

### F — 视觉审美与网站风格总监
- 结论：当前 demo 的审美方向可继续强化，不能回退成数据产品。
- 关键判断：仓库现在已经有暖纸色、柔和面板、绘本标题、轻浮层，不应被 v3.1 的“更多 bubble、更多在线数据、更多分类 icon”稀释成杂乱散点图。
- PRD3.2 指令：审美优先级高于功能扩张；任何新增图层都必须服从“绘本地图”整体感。

### G — 后端全栈与平台工程师
- 结论：当前版本没有后端，这不是缺陷，而是 MVP 边界。
- 关键判断：仓库当前没有 `app/api/*`，没有在线数据依赖，因此当前版本可按静态/半静态展示站点治理；若直接引入 v3.1 的在线 API，会把项目性质改成混合应用。
- PRD3.2 指令：主版本不以新增后端为阻塞项；在线 API 单独作为增强轨设计。

### H — 数据工程与科学可信度负责人
- 结论：当前最大数据风险不是“接口坏了”，而是“叙事与证据未显式分层”。
- 关键判断：`timeline.json` 已提供 18 种物种关键年叙事，但缺少“推断 / 历史记录 / 当代观察”标记；若继续直接引入 ALA 当代点位去表达 1770 年，会显著误导。
- PRD3.2 指令：当前版本坚持本地人工整理数据；补证据层标记，不在 MVP 中用当代 occurrence 伪装历史分布。

### I — AI 内容与知识编辑顾问
- 结论：运行时 LLM 不是当前版本的最佳方案。
- 关键判断：这是科普体验网站，不是即时生成内容站。把扩展物种故事实时交给 LLM，会带来准确性、审校、费用、隐私、稳定性四重风险。
- PRD3.2 指令：主版本不依赖运行时 LLM；若未来使用 LLM，只能用于离线草拟，经人工审核后进入静态数据。

### J — 安全与隐私工程师
- 结论：当前最需要管控的是“未来风险提前入场”。
- 关键判断：当前 demo 没有用户输入上送第三方，也没有公开 API 面，安全面很小；一旦照 v3.1 接入 ALA 代理和 DeepSeek，风险面会陡增。
- PRD3.2 指令：安全设计按“两轨制”写入：当前版本轻量治理，增强轨强制 rate limit、参数校验、隐私说明。

### K — QA、无障碍与兼容性工程师
- 结论：当前代码已有基础 a11y 意识，但远未达到发布级。
- 关键判断：时间轴 slider 有 aria，搜索与按钮也有基础 label，reduced motion 也被引入；但移动端 panel、音频替代说明、WebGL 降级、键盘流与焦点次序仍不完整。
- PRD3.2 指令：将 a11y 从“局部属性”提升为“发布验收项”。

### L — DevOps 与发布工程师
- 结论：当前主版本的部署策略应该先简单化。
- 关键判断：因为当前版本无后端，可优先按静态展示站点治理；但地图底图是远程资源，媒体资产又为空，因此“可构建”不等于“可上线”。
- PRD3.2 指令：把资产预检、环境说明、地图 provider 说明、发布前检查加入强约束。

### M — 产品经理
- 结论：当前版本的真正 MVP 不是“数据更多”，而是“体验完成”。
- 关键判断：对于这个项目，首屏气质、时间探索、区域聚焦、物种故事、音频说明、轻松温和的风格一致性，价值高于扩展到 100-150 种物种。
- PRD3.2 指令：先把 18 种核心物种做到完整可信且有记忆点，再谈扩展库。

### N — PMO 与项目控制负责人
- 结论：`v3.1` 的任务编排不适合作为当前 branch 的交付计划。
- 关键判断：当前代码已具有可演示基础，最优路线不是开启超长数据准备流水线，而是基于已有代码做迭代封板。
- PRD3.2 指令：重排阶段，先做资产、布局、内容与验收，再做增强轨。

### O — 教育传播与投入产出顾问
- 结论：当前项目的最佳收益路径是“高完成度展示型科普网页”，不是“低可信度大而全平台”。
- 关键判断：对一个以展示、探索、教育传播为核心的项目，投入产出最高的工作是：视觉完成度、信息可信度、访问稳定性、可分享性。
- PRD3.2 指令：优先投资高 ROI 项，如媒体资产、移动端、引用与归因、科普文本审校；低 ROI 项如运行时 LLM 与超大扩展物种库后置。

## 2. 对 expert review 的复核结论

### 2.1 直接采纳

以下结论在当前项目中仍然成立，必须写进 `PRD v3.2`：

- `v3.1` 中大量实现级伪代码存在自相矛盾或不完整问题，不适合继续作为直接编码稿。
- 资产存在性是当前最大现实风险之一：`public/assets/species`、`public/assets/audio` 目录存在，但当前为空。
- 音频版权归属必须进入数据结构与 UI。
- 历史叙事与当代数据不能混用而不加说明。
- 移动端 / 平板规格必须补齐。
- 发布平台、环境变量、资源依赖、媒体文件策略必须明确。
- 若未来引入在线 API，输入校验、速率限制、SSE 解析、缓存键、隐私说明都必须明确。

### 2.2 采纳但修正

以下 expert review 观点有价值，但需要按当前仓库现实修正：

- **“Intro 丢失”**：不成立于当前代码。  
  当前仓库已有 [`components/IntroSequence.tsx`](/Users/yichizhang/code/animalmaps/components/IntroSequence.tsx)。  
  正确表述应为：**Intro 在代码里存在，但在 v3.1 的执行逻辑中没有被妥善继承和规格化。**

- **“整个项目没有 a11y 规格”**：不完全成立。  
  当前代码有 aria、keyboard slider、reduced motion 的局部实践。  
  正确表述应为：**项目已有零散可访问性实现，但未形成发布级规范与验收标准。**

- **“移动端完全没有处理”**：不完全成立。  
  当前地图 camera padding、部分 overlay 宽度有响应式考虑。  
  正确表述应为：**移动端已有局部适配，但核心的 `InfoPanel` 仍未形成可用的小屏方案。**

- **“当前项目尚未形成体验主线”**：不成立。  
  当前体验主线已经存在：Intro → 全国态 → 时间轴 → 区域聚焦 → 物种详情。  
  正确表述应为：**体验主线存在，但发布闭环未完成。**

### 2.3 不纳入当前 MVP 的结论

以下内容不是“错”，但不应继续主导当前版本：

- 围绕 `useALASpecies`、LRU、`unstable_cache`、在线 occurrence 代理、在线 LLM route 的大量细节争议。
- 围绕 DOM cloud layer 的性能争议。
- 围绕 100-150 种扩展物种的 bubble 体系设计。
- 围绕运行时 DeepSeek 内容流的产品化方案。

原因不是这些问题不重要，而是：**它们属于增强轨，不应倒逼当前 demo 的 MVP 重新定义自己。**

### 2.4 本轮新增结论

在 expert review 之外，本轮新增以下现实判断：

- 当前 demo 的真正阻塞项是**资产缺失**，不是构建失败。  
  当前 `npm run build` 和 `npm run lint` 均通过。

- 当前 demo 的真正风险是**“看起来像快完成了”，但上线后会出现大量 404 媒体资源与内容归因缺失”**。

- 当前 demo 代码已经具备可持续迭代能力，因此 `PRD v3.2` 应优先规范“如何沿现状增量演进”，而不是继续发明大块未落地子系统。

## 4. PRD v3.2 的产品目标与范围

### 4.1 当前版本目标

当前版本的目标不是做成数据平台，而是把现有 demo 打磨成一个：

- 可以公开演示
- 体验连贯
- 风格统一
- 信息可信
- 资源完整
- 可在桌面与移动端基本可用

的单页科普互动网页。

### 4.2 当前版本的 MVP 定义

当前 MVP 仅围绕**核心 18 种物种**建立完整体验闭环：

1. Intro
2. 全国态地图
3. 区域聚焦
4. 时间轴演化
5. 核心物种详情
6. 经审校的本地故事文本
7. 正确标注的音频 / AI 模拟提示
8. 有归因的媒体资源
9. 基础移动端适配

### 4.3 当前版本明确不以这些能力为上线前置

以下内容不再作为当前 MVP 的阻塞项：

- 扩展到 100-150 种物种
- ALA 运行时 occurrence 查询
- DeepSeek 运行时故事生成
- 地貌环境音自动判定
- DOM cloud bubble layer
- 三态 `InfoPanel` 的扩展物种态

### 4.4 增强轨定义

为避免未来再次混淆，本 PRD 明确采用双轨：

- **主轨 Release A：Demo-to-MVP**
  - 目标：完成当前核心体验
  - 数据：本地静态
  - 发布：以展示站点为主

- **增强轨 Release B：Online Enrichment**
  - 目标：在 Release A 稳定后再做在线丰富化
  - 数据：可接入 ALA、扩展物种、缓存与审核流程
  - 前提：主轨验收完成、内容与隐私策略落地

## 5. 体验与设计规范

### 5.1 视觉总原则

视觉锚点不是 `v2.2`，而是**当前 demo 已经建立的绘本界面语言**：

- 暖纸色背景
- 温和地球色调
- 低攻击性的高亮色
- 浮层像纸片，而不是玻璃或面板系统
- 地图始终是主舞台
- 动效轻、柔、克制，不机械、不果冻化

### 5.2 明确禁止的方向

- SaaS 后台风
- 大数据看板风
- 赛博 / 纯黑科技风
- 过度密集的分类 icon 散点图
- 为了“信息更多”牺牲地图整体呼吸感

### 5.3 页面层级

页面固定保持五层：

1. 地图主舞台
2. 顶部叙事与搜索浮层
3. 图例
4. 右侧 / 底部详情层
5. 底部时间轴

任何新增能力都不能打破“地图是主舞台”的秩序。

### 5.4 Intro 规格

保留当前 `IntroSequence.tsx`，但规范如下：

- 总时长不超过 9 秒
- 首次访问自动播放
- 支持跳过
- 二次访问默认不重复播放
- 文案只承担两件事：
  - 建立“翻开一张自然绘本地图”的气质
  - 教会用户“拖动时间，看见生命故事”

### 5.5 顶部浮层规格

`TopOverlay.tsx` 保留当前结构，补齐以下约束：

- 搜索框在当前 MVP 中只搜索核心 18 种物种。
- 搜索 placeholder 必须明确：`搜索物种名称`
- 下拉结果最多显示 6 项。
- 无结果时保留当前柔和空态，不显示报错语气。
- 区域聚焦时“返回全局”按钮必须始终可见。

### 5.6 详情层规格

当前版本只保留两态：

- `region`
- `core-species`

不在主轨内引入 `extended-species` 第三态。

桌面端规格：

- `>= 1280px`：固定右侧 panel，宽 `400px`
- `768px - 1279px`：固定右侧 panel，宽 `360px`
- `< 768px`：改为底部 drawer，宽 `100%`，最大高度 `68vh`

移动端 drawer 必须满足：

- 可上滑查看更多
- 不遮死顶部搜索与核心地图点击
- 关闭按钮可见
- 返回区域按钮在物种态下可见

### 5.7 时间轴规格

保留当前 `TimelineBar.tsx` 的交互方式，补充以下约束：

- 拖动必须继续支持键盘操作。
- 时间标签继续保留 `推断层 / 事实层 / 探索层`。
- 当前 MVP 不增加更多节点，只稳定现有节点。
- 详细态下时间轴缩窄不能影响 390px 手机可操作性。

### 5.8 地图与 marker 规格

当前版本的主原则是：

- 先保证 marker 系统稳定、审美一致、不会视觉爆炸。
- 当前主轨不启用在线多点分布气泡系统。
- 当前主轨允许继续使用“每个核心物种一个主 marker”的表达。

允许的升级方向：

- 在核心 18 种媒体资产到位后，把主 marker 从“分组 icon”升级为“物种插图头像”。
- 若插图未就位，必须保留当前通用 icon fallback，不得出现断图或空白。

### 5.9 空态与异常态

必须设计以下 UI：

- 音频文件不存在时：显示“暂无音频”，不显示坏掉的播放器
- 搜索无结果时：保留温和引导语
- 地图底图失败时：显示地图加载提示，不要只剩空白区域
- 物种媒体未准备好时：使用品牌色占位图形，不显示 broken image

## 6. 数据与内容规范

### 6.1 当前版本数据原则

当前版本只使用**本地静态数据**作为前台展示真相来源。

原因：

- 与当前代码现状一致
- 更适合 agent 稳定开发
- 便于内容审校
- 便于发布控制
- 避免把科普产品过早变成实时数据应用

### 6.2 `species.json` 规范

继续保留当前核心字段，但必须新增或补齐以下信息：

- `assetStatus`
  - `placeholder`
  - `partial`
  - `complete`
- `media`
  - `illustration`
  - `photo`
  - `audio`
- `sources`
  - 至少 1-3 条可信来源
- `reviewStatus`
  - `draft`
  - `reviewed`
  - `approved`

### 6.3 `timeline.json` 规范

当前版本必须给关键时间叙事新增可信度分层，推荐结构：

- `evidenceType`
  - `inferred`
  - `historical`
  - `contemporary`

这用于明确区分：

- 1770 / 1788 前后的推断性叙事
- 殖民后有记录支持的历史叙事
- 当代状态

### 6.4 `audio.json` 规范

除现有字段外，必须新增：

- `attribution`
- `license`
- `sourceUrl`
- `availability`

其中：

- 已灭绝物种的 AI 模拟音频必须显示“非真实历史录音”
- 所有可播放音频必须可追溯来源

### 6.5 引用与归因原则

当前版本必须把“可追溯性”视为发布条件：

- 音频有归因
- 照片有归因
- 关键科学叙事有来源
- AI 模拟内容必须标注

### 6.6 当前版本不采用的做法

当前版本不采用：

- 用 ALA 当代点位直接代表 1770 年空间分布
- 把运行时 LLM 输出直接作为核心科普内容
- 在未审校前把自动生成文本直接上站

## 8. 后端、接口与在线增强轨

### 8.1 主轨结论

当前主轨 **不要求后端**。

理由：

- 与当前代码一致
- 最小化风险
- 更利于发布
- 不让 agent 在当前版本被 API 细节牵制

### 8.2 增强轨允许的能力

仅在 Release A 验收后，增强轨才允许讨论：

- `GET /api/ala/occurrences`
- `GET /api/ala/species`
- `POST /api/llm/story`
- `hooks/useALASpecies.ts`

### 8.3 增强轨强制前提

若未来进入增强轨，以下要求是强制项：

- 部署平台明确为 Vercel Node.js runtime
- 所有 API route 有输入校验
- 所有公开写路径有 rate limit
- LLM 输出仅作文本显示，禁止 HTML 注入
- SSE 解析需处理跨 chunk
- 缓存 key 包含完整参数
- 有隐私说明页
- 在线生成内容默认不进入“核心科普事实层”

### 8.4 增强轨的数据原则

增强轨也不得绕过科学性：

- ALA 点位只能作为“观察记录”或“补充参考”
- 历史分布若缺乏高质量证据，不得冒充精确地图事实
- LLM 生成内容只能作为辅助叙事，不能取代经过审校的核心内容

## 9. 安全、合规与可信度要求

### 9.1 当前主轨必须满足

- 不出现 `dangerouslySetInnerHTML` 渲染外部文本

### 10.1 当前版本阶段重排

- 补齐 `.env.example`

- 引用与归因

## 13. 最终结论

`PRD v3.2` 的核心不是继续给当前项目叠加“更大、更在线、更自动化”的愿景，而是明确：

- 现在的 demo 已经长出了正确的骨架
- 现在最该做的是把这副骨架长成一个完整、可信、优雅、可发布的科普作品
- 真正的基线不是旧 PRD，而是当前仓库里已经存在、已经能 build、已经能讲故事的这套代码

因此，本版 PRD 的立场非常明确：

**先把当前 demo 打磨成高完成度的 18 物种绘本科普体验，再决定是否进入在线增强轨。**
